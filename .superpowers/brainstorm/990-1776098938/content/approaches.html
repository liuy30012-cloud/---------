图书损坏报告 - 实现方案对比

三种架构方案，请评估权衡后选择

A

方案A：独立损坏报告模块（推荐）

架构：新建独立的 DamageReport 实体 + Controller + Service + 前端页面

  • 独立的 damage_reports 数据表（bookId, userId, photos, type, description, status）
  • 独立的 DamageReportController（/api/damage-reports）
  • 独立的 DamageReportService 处理业务逻辑
  • 前端：BookDetail 页内嵌"报告问题"弹窗 + 独立的"我的报告"页面
  • 照片存储：本地 uploads/damage-photos/ 目录
  • 与现有通知系统联动

优点

  • 完全解耦，不影响现有代码
  • 可独立扩展（如日后加入扫码报修）
  • 符合项目现有的模块化架构风格
  • 测试和调试简单

缺点

  • 新建文件较多（约15个文件）
B

方案B：扩展 BookReview 模式

架构：在现有评价系统基础上扩展"问题报告"类型

  • 给 BookReview 表添加 reviewType 字段（REVIEW / DAMAGE_REPORT）
  • 扩展 BookReviewController 添加损坏报告相关端点
  • 复用前端评论组件框架，切换为报告表单
  • 照片存储：本地目录

优点

  • 改动文件少（约8个文件）
  • 复用已有基础设施

缺点

  • 评价和损坏报告语义完全不同，强行合并违反单一职责
  • 评论查询需要到处加 type 过滤
  • 日后两个功能各自扩展时会互相干扰
  • 数据库查询效率下降（混合数据）
C

方案C：利用 BookTag 标签系统

架构：通过标签标记问题书籍，用独立表存储报告详情

  • 复用 BookTag 系统添加 SYSTEM 标签（如 DAMAGED、UNDER_REPAIR）
  • 新建轻量 DamageReport 表只存详情
  • 标签变更自动关联报告

优点

  • 利用现有标签基础设施
  • 问题书籍可通过标签系统被搜索到

缺点

  • 标签是静态标记，无法表达复杂的报告状态流转
  • 标签系统目前前端没有管理 UI
  • 标签和报告的关联逻辑复杂
  • 过度依赖标签系统限制了报告功能扩展

推荐：方案A

方案A 虽然新建文件多，但完全遵循项目现有的模块化架构（每个功能域一个独立的 Model + Repository + Service + Controller + DTO + 前端API），与其他模块（借阅、预约、评价、采购）风格一致。方案B的"评价兼报告"会让代码变得混乱，方案C的标签方案过于简陋。
